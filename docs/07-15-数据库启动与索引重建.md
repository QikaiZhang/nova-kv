# 数据库启动与索引重建设计

> 本文记录 nova-kv 数据库启动流程的设计决策，以 Maintainer 视角解释"为什么这样设计"，而非"代码做了什么"。

---

## 一、背景

nova-kv 采用 Bitcask 存储模型：磁盘上存 append-only 数据文件，内存中存索引（key → 文件位置）。这种架构下，启动流程是整个系统的关键路径——它决定了数据库能否正确恢复、数据是否完整。

启动时要做的事其实很简单：

1. 校验配置
2. 打开数据目录
3. 扫描所有 `.data` 文件
4. 逐条读取记录，重建内存索引
5. 返回可用的 DB 实例

但每一步都有设计决策需要权衡。下面逐个分析。

---

## 二、配置校验：为什么要在 Open 一开始就做？

### 当前方案

在 `Open()` 最开始调用 `validateOptions()`，检查：

- `DirPath` 非空
- `DataFileSize > 0`
- `IndexType` 为已知类型

校验通过后才执行 `MkdirAll` 等磁盘操作。

### 为什么这个顺序重要？

**校验失败时，不应产生任何磁盘副作用。** 如果用户传了空目录路径就调用了 `Open`，创建了一个叫 `""` 的目录就太荒谬了。Fail fast 原则——先拒绝非法输入，再做事。

### 为什么 IndexType 用 `int8` 而不是 string？

- **编译期安全**：`int8` + 常量，编译器能检查。string 只能运行时检查
- **零值可检测**：`iota + 1` 使零值（0）不是有效类型，防止"忘了设置"的 bug
- **够用**：索引类型不会超过 127 种

### 为什么 DataFileSize 只检查 `> 0` 而不设更严格的下限？

当前只检查 `DataFileSize > 0`，不强制最小值（如 1MB）。原因：

- 项目处于早期阶段，测试时可能需要极小的文件阈值来触发轮转
- 生产环境中，用户自行负责合理配置
- 过早限制会妨碍灵活性

后续可考虑在文档中建议最小值，但不硬编码限制。

## 三、索引类型选择：为什么用 switch 而不是工厂模式？

### 当前方案

```go
switch options.IndexType {
case BTree:
    idx = index.NewBTree()
default:
    return nil, ErrOptionsIndexTypeUnknown
}
```

### 为什么不用注册表/工厂模式？

项目目前只有一个索引实现（BTree）。注册表模式（`map[string]NewIndexFunc`）的收益在当前规模下为零，但增加了：

- 全局状态（init 注册）
- 反射/字符串开销
- 包依赖复杂度

**switch 的缺点是添加新索引类型要改 Open()，但这正是当前项目需要的——让变更显式、可控。** 等到有 3+ 种索引实现时再考虑工厂模式。

### 为什么当前只支持 BTree？

BTree 是平衡选择：

- **查询性能**：O(log N)，排序键范围查询友好
- **内存效率**：节点利用率高，不像 HashMap 有 rehash 抖动
- **成熟实现**：`google/btree` 经过充分测试

后续可选方案（已预留 IndexType 扩展点）：

| 索引 | 读性能 | 写性能 | 内存占用 | 适用场景 |
|------|--------|--------|----------|----------|
| BTree | O(log N) | O(log N) | 中 | 通用 |
| ART | O(k) | O(k) | 低 | key 较长 |
| HashMap | O(1) | O(1) | 高 | 纯点查 |
| SkipList | O(log N) | O(log N) | 中 | 并发写友好 |

## 四、文件发现与加载：命名约定与容错

### 文件命名约定

nova-kv 使用 9 位零填充数字命名数据文件：`000000001.data`、`000000002.data`……

选择 9 位的理由：

- **排序友好**：字符串排序 = 数字排序，不需要额外逻辑
- **够用**：9 位支持 10 亿个文件，每个 256MB = 256PB 总容量，远超单机需求
- **可读**：一眼看出是第几个文件

为什么不学 LevelDB 用 `.sst`、`.log`、`.MANIFEST` 等多后缀？

- Bitcask 模型不需要区分文件类型——所有数据文件格式完全相同
- 单后缀 `.data` 足够简单

### 文件 ID 间隔容忍

`loadDataFiles()` 不要求文件 ID 连续。例如 `[1, 5, 9]` 是合法的。

**为什么允许间隔？**

- Merge 操作会删除旧文件，自然产生间隔
- 手动清理也可能遗留间隔
- 强制连续性增加了不必要的约束

**Active File 判定**：ID 最大的文件 = 活跃文件。简单直观，不需要额外标记。

### 文件名解析加固

当前实现用 `TrimSuffix(name, ".data")` 去掉后缀，然后验证：

```
1. 剩余部分必须恰好 9 个字符
2. 必须能解析为非负整数
```

这样 `1.data`、`abc.data`、`000000abc.data`、`-000000001.data` 都会被正确过滤。

**为什么不直接用 `strings.Split` 取 `[0]`？**

原实现 `strings.Split("abc.data", ".")` 得到 `["abc", "data"]`，`Atoi("abc")` 失败后静默跳过——逻辑上正确但不够严谨。新实现明确表达了"必须 9 位数字"的意图。

## 五、索引重建：核心设计决策

### 5.1 为什么每次启动都全量扫描？

**RocksDB 有 MANIFEST，BadgerDB 有 Value Log + SST，为什么 nova-kv 不做 checkpoint？**

因为 Bitcask 模型的索引重建成本与数据量成正比，但收益是极致的简单性：

- **没有元数据文件需要维护**：无需 MANIFEST、WAL、checkpoint 之间的复杂一致性
- **磁盘是唯一真相来源**：不会出现"MANIFEST 和实际文件不一致"这类 bug
- **启动恢复逻辑只有 60 行**：相比之下 RocksDB 的恢复逻辑有数千行

**代价是启动速度与数据量线性相关。** 这是有意为之的 trade-off：

> 用"启动时的一次性扫描成本"换取"运行时的极简写路径"。

### 5.2 为什么从最老的文件处理到最新？

```
文件 1: key1=v1
文件 2: key1=v2  ← 更新版本
文件 3: key1=v3  ← 最新版本
```

按文件 ID 从小到大处理：同一 key 被多次 Put 时，后处理的（新值）覆盖先处理的（旧值），最终索引指向最新值。

**如果反过来（从新到旧）会怎样？**

需要判断"这个 key 已经在索引中了，跳过"。多了一次判断，但结果相同。所以从旧到新不是唯一正确的方式，但实现更直接——不需要检查 key 是否已存在。

### 5.3 损坏记录处理：为什么跳过整个文件？

遇到无法解码的记录时，**放弃当前文件的剩余部分**，继续处理下一个文件。

**为什么不是逐字节跳跃尝试恢复？**

逐字节跳跃寻找下一条有效记录的风险：

- 可能在损坏点之后"碰巧对齐"出假记录 → **假数据进入索引**
- 这是比丢失数据更严重的问题——用户会读到错误的数据而不知情

**为什么不是直接 panic 或拒绝启动？**

一个 bit-flip 不应让整个数据库无法启动。保守策略：

- 损坏文件之前的所有文件：完整恢复
- 损坏文件：恢复到损坏点之前
- 损坏文件之后的所有文件：完整恢复

**后续 CRC 校验实现后可以更精细地处理**——能确定性地跳过单条损坏记录，而不是整个文件。

### 5.4 未知记录类型的向前兼容

对 `LogRecordNormal` 和 `LogRecordDeleted` 之外的类型静默忽略。

这是为了向前兼容：未来可能新增记录类型（如 `LogRecordMergeHint`、`LogRecordCheckpoint` 等），旧版本数据库启动时不认识新类型，应该跳过而不是崩溃。

## 六、与业界方案对比

| 引擎 | 恢复方式 | 启动速度 | 复杂度 |
|------|----------|----------|--------|
| **Bitcask** | 全量扫描数据文件 | O(数据量) | 极低 |
| **LevelDB/RocksDB** | MANIFEST + WAL replay | O(WAL 大小) | 高 |
| **BadgerDB** | Manifest + Value Log 扫描 | O(Value Log) | 中 |
| **BoltDB** | mmap B+Tree，直接可用 | O(1) | 中 |
| **nova-kv** | 全量扫描 .data 文件 | O(数据量) | 极低 |

**nova-kv 选择 Bitcask 模式的原因：**

- 项目目标是**学习和展示存储引擎设计思想**，不是生产级性能
- 简单性本身就是价值——代码行数少，容易理解和修改
- 全量扫描在中小规模数据下完全可接受（256MB 文件读取只需几秒）

## 七、当前限制与后续优化

### 已明确不实现（当前阶段）

| 特性 | 原因 |
|------|------|
| CRC 校验 | 编码已预留字段，等流程跑通后再补 |
| Hint File | 需要先实现 Merge，Merge 是更大的工程 |
| 并发恢复 | 启动不是热路径，单线程够用 |
| 增量 checkpoint | 引入元数据管理，破坏"磁盘即真相"的简单性 |

### 后续可优化

1. **Hint File**：Merge 时生成，启动时优先加载，跳过已有索引的 key
2. **CRC 校验**：header 的 4 字节 CRC 字段已在编码格式中预留
3. **恢复进度回调**：大数据量时提供进度百分比
4. **Checksum File**：记录文件级 checksum，快速判断文件是否完整

## 八、文件轮转的 Appender 语义修复

在本次实现中发现了一个预存在的 bug：

**问题**：`fio.FileIO` 打开文件时未使用 `O_APPEND` 标志。文件关闭再重新打开后，光标回到位置 0，`Write()` 从文件开头覆盖写入。

**修复**：`os.OpenFile(path, os.O_RDWR|os.O_CREATE|os.O_APPEND, 0644)`

`O_APPEND` 保证每次 `Write()` 都追加到文件末尾，无论当前光标位置。这对 Bitcask 的 append-only 语义是必需的。

**不使用 `O_APPEND` 的替代方案**：手动 `Seek(0, io.SeekEnd)` 在每次写入前定位到末尾。但 `O_APPEND` 更简洁且是原子操作（内核保证），不会有 TOCTOU 问题。

### ReadLogRecord 的 EOF 处理修复

**问题**：`os.File.ReadAt` 在读到文件末尾时，会同时返回读取的数据和 `io.EOF`。原代码将任何 error 都视为失败，导致最后一条记录被丢弃。

**修复**：区分"读到 0 字节 + io.EOF"（真正的文件末尾）和"读到 n 字节 + io.EOF"（读到了最后一条记录的末尾）。后者应正常解码处理。

## 十、建议手写代码详解

以下是代码中标记了 `Strongly Recommended to Handwrite` 的核心逻辑，逐段讲解为什么要手写、每一步在干什么、容易错在哪里。

---

### 10.1 `validateOptions` — 启动校验

```go
// Strongly Recommended to Handwrite
func validateOptions(opts *Options) error {
    if opts.DirPath == "" {
        return ErrOptionsDirPathEmpty
    }
    if opts.DataFileSize <= 0 {
        return ErrOptionsDataFileSizeTooSmall
    }
    if opts.IndexType != BTree {
        return ErrOptionsIndexTypeUnknown
    }
    return nil
}
```

**为什么这段必须手写？**

这是数据库的第一道防线。如果不校验：

- `DirPath=""` → `MkdirAll("")` 行为未定义
- `DataFileSize=0` → 每条记录都触发文件轮转，日志文件爆炸
- `IndexType=0` → 零值静默通过，用户不知道索引没被正确初始化

**校验顺序为什么这样排？**

1. DirPath 为空 → 最基础的配置错误，先查
2. DataFileSize ≤ 0 → 语义错误，其次
3. IndexType 未知 → 类型系统保证不了，运行时检查

**为什么用 `<=` 而不是 `<`？**

`DataFileSize == 0` 是合法编译但完全不可用的配置。如果 DataFileSize = 0，`WriteOff + size > 0` 对任何非空记录都成立，每条记录都触发轮转。尽早拒绝。

**为什么不设最小值（如 1MB）？**

见前文 §二 的讨论——早期项目保持灵活，测试时可能需要极小阈值。

---

### 10.2 `loadDataFiles` — 文件发现与加载

```go
// Strongly Recommended to Handwrite
func (db *DB) loadDataFiles() error {
    dirEntries, err := os.ReadDir(db.options.DirPath)
    // ...
    for _, entry := range dirEntries {
        if !strings.HasSuffix(entry.Name(), data.DataFileNameSuffix) {
            continue
        }
        trimmed := strings.TrimSuffix(entry.Name(), data.DataFileNameSuffix)
        if len(trimmed) != 9 {
            continue
        }
        fileId, err := strconv.Atoi(trimmed)
        if err != nil || fileId < 0 {
            continue
        }
        fileIds = append(fileIds, fileId)
    }
    sort.Ints(fileIds)
    // ...
}
```

**为什么要 TrimSuffix + 长度校验，而不是 Split(".")？**

原实现：
```go
splitNames := strings.Split("my.backup.data", ".")
fileId, _ := strconv.Atoi(splitNames[0]) // splitNames[0] = "my"
```
`Atoi("my")` 失败，`continue`。逻辑对，但不够显式。

新实现：
```go
trimmed := strings.TrimSuffix("my.backup.data", ".data") // "my.backup"
len(trimmed) != 9 → continue // 明确拒绝
```

**对比**：TrimSuffix 精确去掉 `.data` 后缀，长度校验 `!= 9` 直接表达"必须 9 位数字"的意图，比 Split 取 [0] 更显式。

**为什么文件 ID 不要求连续？**

见前文 §四——Merge、手动清理都会留间隔。代码层面 `sort.Ints(fileIds)` 按 ID 排序即可，不需要额外的连续性检查。

**为什么最大 ID = 活跃文件？**

Bitcask 模型的性质：新文件 ID 递增（`activeFile.FileId + 1`），所以最大 ID 一定是最后创建的文件 = 活跃文件。不需要额外的状态标记。

**为什么用 `os.Stat` 设置 WriteOff？**

文件关闭再打开后，`WriteOff = 0`。如果不用 `Stat().Size()` 恢复，后续 `Write()` 会从 0 开始覆盖（虽然有 `O_APPEND` 保护，但 WriteOff 还需要用于文件轮转判断和索引位置计算）。

### 10.3 `loadIndexFromDataFiles` — 索引重建

```go
// Strongly Recommended to Handwrite
func (db *DB) loadIndexFromDataFiles() error {
    // 收集文件 ID，排序
    // ...
    for _, fid := range fileIds {
        var offset int64 = 0
        for {
            logRecord, size, err := dataFile.ReadLogRecord(offset)
            if err != nil {
                if err == io.EOF {
                    break // 当前文件读完，正常结束
                }
                // 非 EOF = 记录损坏
                corruptedRecords = append(corruptedRecords, &CorruptedRecordError{...})
                break // ← 关键：跳过当前文件剩余，继续下一个
            }
            // 更新索引
            if logRecord.Type == data.LogRecordNormal {
                db.index.Put(logRecord.Key, pos)
            } else if logRecord.Type == data.LogRecordDeleted {
                db.index.Delete(logRecord.Key)
            }
            offset += size
        }
    }
    // 返回第一个损坏错误（供上层日志/监控）
}
```

**处理顺序为什么从小到大？**

```
文件 1 (最老): key1=v1
文件 2:        key1=v2  ← 更新
文件 3 (最新): key1=v3  ← 最新
```

从小到大遍历：v1 → v2 → v3，最终索引指向 v3。天然正确，不需要额外判断"key 是否已存在"。

**为什么损坏时 break 而不是 continue？**

这是整段代码最容易出错的地方。见前文 §5.3 的详细分析：

- **逐字节跳跃** = 可能伪造假记录，比丢失数据更危险
- **break（跳过文件剩余）** = 保守但安全，只在"丢失数据"上付出代价
- **return err（拒绝启动）** = 一个 bit-flip 让整个数据库不可用，不合理

当前选 break。等 CRC 实现后可在确定损坏边界的情况下精确跳过单条记录。

**为什么对未知记录类型静默忽略？**

向前兼容。如果 v1 版本写入的记录类型在未来被 v2 识别，v1 重启时不应崩溃。静默跳过是安全的选择——不读总比读错了强。

**CorruptedRecordError 为什么实现 `Unwrap()`？**

```go
type CorruptedRecordError struct {
    FileId uint32
    Offset int64
    Cause  error
}
func (e *CorruptedRecordError) Unwrap() error { return e.Cause }
```

调用方可以用 `errors.As` 提取原始错误：
```go
var corrupted *CorruptedRecordError
if errors.As(err, &corrupted) {
    log.Printf("文件 %d 偏移 %d 损坏: %v", corrupted.FileId, corrupted.Offset, corrupted.Cause)
}
```

这在生产环境排查问题时非常有用。

### 10.4 `ReadLogRecord` — 记录读取与 EOF 处理

```go
// Strongly Recommended to Handwrite
func (df *DataFile) ReadLogRecord(offset int64) (*LogRecord, int64, error) {
    headerBuf := make([]byte, maxLogRecordHeaderSize)
    n, err := df.IoManager.Read(headerBuf, offset)

    if err == io.EOF && n == 0 {    // ← 关键判断 1
        return nil, 0, io.EOF
    }
    if err != nil && err != io.EOF { // ← 关键判断 2
        return nil, 0, err
    }
    // 此时: err == nil 或 err == io.EOF（但 n > 0）
    // 继续解码...
}
```

**为什么这段代码看起来绕来绕去的？**

Go 的 `io.ReaderAt.ReadAt` 有个反直觉的行为：

> 当读取范围超过文件末尾时，**同时返回已读取的字节和 `io.EOF`**。

也就是说，如果文件只有 39 字节，你从 offset=26 请求读 21 字节：

- 实际读取：13 字节（offset 26 到 38）
- 返回：`n=13, err=io.EOF`

原代码的逻辑是：
```go
if err != nil { return nil, 0, err } // ← 把"读到数据 + EOF"当成了错误！
```
这导致最后一条记录（刚好在文件末尾）被丢弃，索引重建时永远少一条。

**修复后的判断逻辑：**

1. `err == io.EOF && n == 0` → 真·文件末尾，正常返回 EOF
2. `err != nil && err != io.EOF` → 真正的 I/O 错误，返回错误
3. `err == nil` 或 `err == io.EOF 且 n > 0` → 读取了有效数据，继续解码

步骤 3 是关键——即使 `err == io.EOF`，只要读到了字节，就当作有效数据处理。

**为什么 `maxLogRecordHeaderSize = 21` 而不是读精确长度？**

因为头部是变长的——`KeySize` 和 `ValueSize` 用 varint，不读出来不知道具体多长。21 字节是理论最大值：

```
CRC(4) + Type(1) + KeySize(max 8) + ValueSize(max 8) = 21
```

用最大值读一次，再根据实际解码出的 header 读 key/value，比"读 Type → 读 KeySize → 读 ValueSize"三次 IO 更高效。

### 10.5 `EncodeLogRecord` — 二进制编码

虽然不属于本次启动模块，但同样标记了 `Strongly Recommended to Handwrite`，一并讲解：

```go
// log_records.go
func EncodeLogRecord(logRecord *LogRecord) ([]byte, int64) {
    header := make([]byte, maxLogRecordHeaderSize)

    header[4] = logRecord.Type  // CRC 占 0-3，Type 在 byte 4

    var index = 5
    index += binary.PutVarint(header[index:], int64(len(logRecord.Key)))
    index += binary.PutVarint(header[index:], int64(len(logRecord.Value)))

    var size = index + len(logRecord.Key) + len(logRecord.Value)
    encBytes := make([]byte, size)
    copy(encBytes[:index], header[:index])
    copy(encBytes[index:], logRecord.Key)
    copy(encBytes[index+len(logRecord.Key):], logRecord.Value)
    return encBytes, int64(size)
}
```

**为什么 CRC 字段预留了但不填？**

编码格式中前 4 字节是 CRC32，当前写为 0：
```
+-------+--------+-----------+-------------+-----+-------+
| CRC   |  Type  |  KeySize  |  ValueSize  | Key | Value |
+-------+--------+-----------+-------------+-----+-------+
  4B      1B       varint       varint       变长   变长
```

现在不填的理由：
1. 先让核心流程跑通（写入、读取、索引恢复）
2. 解码端同样跳过 CRC 校验，预留的 4 字节写成 0，不影响读取
3. 后续一个 PR 即可完成 CRC 实现（写入+校验），不破坏现有格式

**为什么用 `binary.PutVarint` 而不是固定 `uint32`？**

固定 4 字节的 `uint32` 对于 key size = 5 这种小 key 浪费了 3 字节。Varint 对小值用 1 字节，对 100KB 的大值才用 4 字节。Bitcask 的键通常很短（几十字节），Varint 节省的空间可观。

**为什么先写 header+key+value 到一个大数组再分割，而不是逐段分配？**

减少内存分配次数。一次 `make([]byte, size)` 分配一块连续内存，三次 `copy` 是 O(n) 的 memcpy。比 `append(append(header, key...), value...)` 少两次扩容。
