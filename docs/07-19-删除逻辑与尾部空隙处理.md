# 删除逻辑与文件尾部空隙处理

> 本文分析 nova-kv 中 Delete 操作的三个阶段、文件尾部残缺字节（tail gap）的成因与风险，以及修复方案。

---

## 一、Bitcask Delete 的三阶段分工

Bitcask 的 Delete 不是"删磁盘数据"，而是"写入墓碑 + 移除索引"。真正的空间回收在 Merge 阶段完成。

### 1.1 在线删除 — appendLogRecord + 移除索引

```
Delete(key):
  1. 查内存索引 → 不存在则直接返回（幂等）
  2. 追加墓碑记录: {Key, Value=nil, Type=Deleted} → active file
  3. 从内存索引移除该 key
```

**关键特性**：
- 磁盘数据不删除，只追加墓碑记录
- 索引中移除后，后续 Get 查不到 → 返回 ErrKeyNotFound
- 写入和索引移除不是原子操作：先写磁盘后更新索引，写入成功但索引更新失败时，重启后可恢复（磁盘是真相来源）

**为什么不存在的 key 不追加墓碑？**

幂等性。如果 key 从未存在过、或已被删除后再次 Delete，磁盘上已有（或不需要）墓碑记录。追加无意义的墓碑只会浪费磁盘空间，Merge 时还得额外处理。

### 1.2 重启重建索引 — loadIndexFromDataFiles

```
loadIndexFromDataFiles():
  for 每个数据文件（ID 升序）:
    offset = 0
    while offset < 文件大小:
      record, size = ReadLogRecord(offset)
      if record.Type == Normal:
        index.Put(key, pos)     // 后处理的覆盖先处理的
      else if record.Type == Deleted:
        index.Delete(key)       // 从索引中移除
      offset += size
```

**为什么要从小到大遍历？**

后处理的记录覆盖先处理的结果。如果 key 原本有值，后面跟着墓碑，最终索引中没有该 key——正确表示"已删除"。反过来（先墓碑后正常值），最终索引中保留 key——正确表示"最新是写入"。

### 1.3 Merge 磁盘清理 — 尚未实现

当前项目阶段：append-only，磁盘只增不减。Merge 的设计留到后续版本：

```
Merge():
  1. 遍历所有旧文件（非 active file）
  2. 跳过墓碑记录
  3. 对于同 key 的多条记录，只保留最新的
  4. 写入新的 merge 文件
  5. 删除原始旧文件（已有新文件覆盖）
  6. 更新索引指向新文件位置
```

**Merge 的关键约束**：不能 merge active file（正在写入），只能 merge 已封存的 immutable 文件。

## 二、文件尾部空隙问题

### 2.1 什么是尾部空隙

数据文件在追加写入记录时，如果进程崩溃、断电或磁盘满，最后一条记录可能只写入了一部分：

```
正常文件:  [完整记录 A][完整记录 B][完整记录 C]
                                         ↑ WriteOff

崩溃后:    [完整记录 A][完整记录 B][残缺的 C...]
                                   ↑ 只有半截 header 或半截 key/value
```

这半截字节就是"尾部空隙"（tail gap）。

### 2.2 PUT 和 DELETE 都会触发

**两种操作都通过 `appendLogRecord` 追加写入**，写入中断机制完全相同：

| 操作 | 追加内容 | 中断后果 |
|------|----------|----------|
| PUT | Normal 记录 (Key + Value) | 尾部残缺 Normal 记录 |
| DELETE | 墓碑记录 (Key, 无 Value) | 尾部残缺 Deleted 记录 |

空隙的产生与记录类型无关，只与写入是否完整有关。

### 2.3 当前代码中的隐性 Bug

问题在 `DecodeLogRecordHeader` 中对截断 varint 的处理：

```go
// 原代码
keySize, n := binary.Varint(buf[index:])
index += n
header.keySize = uint32(keySize)
```

**`binary.Varint` 合约**：当输入字节不构成完整 varint 时（最后一个字节 MSB=1 但无后续字节），返回 `(0, 0)`。

原代码没有检查 `n` 是否为 0，直接将 `keySize=0` 当作合法值，导致：

```
残缺尾部字节 → DecodeLogRecordHeader 返回 {recordType=某值, keySize=0, valueSize=0, headerSize=5}
                ↓
ReadLogRecord: kvSize=0, 不读 key/value, 返回一条空记录
                ↓
loadIndexFromDataFiles: Type==Normal → index.Put([]byte{}, pos)  ← 污染索引！
```

**风险**：一个写入中断的 PUT 记录残片，会被当作一条合法的空 key 记录插入内存索引，后续 Get 空 key 会返回垃圾位置。

### 2.4 为什么正常读写不受影响

- **写入**：active file 用 `WriteOff` 追踪当前偏移，不依赖回读。O_APPEND 保证追加到真实 EOF。
- **读取**：Get 只走索引指向的有效 offset。索引中指向的位置都是完整记录写入成功后记录的。
- **重启**：`WriteOff = stat.Size()` 指向真实文件末尾（含空隙），后续写入从正确位置开始。

空隙只在**重启索引重建时**暴露——遍历文件时尝试解码残缺字节。

## 三、修复方案

### 3.1 核心修复：检测截断 varint

在 `DecodeLogRecordHeader` 中，每次 `binary.Varint` 调用后检查返回值 `n`：

```go
keySize, n := binary.Varint(buf[index:])
if n <= 0 {             // ← 新增：varint 截断检测
    return nil, 0
}
index += n
header.keySize = uint32(keySize)

valueSize, n := binary.Varint(buf[index:])
if n <= 0 {             // ← 新增
    return nil, 0
}
index += n
header.valueSize = uint32(valueSize)
```

**效果**：

- `n <= 0` → 返回 `nil, 0` → `ReadLogRecord` 返回 `"decode log record header failed"` → `loadIndexFromDataFiles` 将其记入 `corruptedRecords`，break 跳过当前文件剩余部分
- 尾部空隙不再产生假的空记录，索引不会被污染

### 3.2 错误信息可观测性

增加了 `ErrIncompleteHeader` 导出错误：

```go
var ErrIncompleteHeader = errors.New(
    "log record header is incomplete, possible truncated write at file tail",
)
```

调用方可以通过 `errors.Is(err, ErrIncompleteHeader)` 区分"文件尾部自然空隙"和"真正的数据损坏"，在监控和日志中给予不同的处理级别。

### 3.3 纵深防御：后续 CRC 校验

当前 CRC 字段预留但未实现。加上 CRC 后形成双层防护：

```
第一层: varint 截断检查 → 捕获写入中断的残缺字节
第二层: CRC32 校验      → 捕获磁盘静默损坏（bit rot）
```

完整记录即使 varint 解析通过，CRC 不匹配也能被识别。两者互为补充，覆盖不同的故障模型。

### 3.4 Merge 后的最终回收

尾部空隙的字节在 Merge 时自然消失——Merge 只复制完整记录到新文件，旧文件（含空隙）直接删除，磁盘空间彻底回收。

## 四、设计对照：为什么不像 RocksDB 一样写 MANIFEST

RocksDB 通过 MANIFEST 文件记录 SST 文件列表和元数据，启动时读 MANIFEST 而非全量扫描。为什么 nova-kv 不这样做？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 全量扫描（当前） | 零额外元数据、磁盘是唯一真相来源、实现简单 | 启动时间正比于数据量 |
| MANIFEST（RocksDB） | 启动快、精确知道文件边界 | 需维护额外文件、崩溃时 MANIFEST 自身可能不一致 |

对于单机 KV 存储，全量扫描的启动时间通常可接受（几 GB 数据秒级完成）。MANIFEST 的复杂度在当前阶段不值得引入。
