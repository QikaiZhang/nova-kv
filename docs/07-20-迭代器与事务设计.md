# 索引迭代器与批量事务设计

> 本文面向 nova-kv 的开发者/维护者，解释「为什么要加 Iterator」「WriteBatch 怎么保证原子性」以及「MVCC 为什么暂时不实现」。用大白话，不堆术语。

---

## 一、为什么要搞个 Iterator？

### 问题场景

之前你只能 `Put`、`Get`、`Delete`。需求来了：前端要做个管理后台，展示数据库里所有的 key，或者按前缀 `user:` 列出用户数据。你怎么搞？

只能对着代码发呆——索引全在 BTree 里，但是 BTree 的 `Ascend`/`Descend` 方法藏得老深，外面根本调不到。

### 解决思路

给 `Indexer` 接口加个 `NewIterator(opts)`。BTree 的迭代器实现思路超简单：因为索引全在内存里，Rewind 或 Seek 的时候就一次性把匹配的 item 全部收集到一个 `[]*Item` 切片里，然后一个游标来回走。

**好处**：
- 遍历过程中不需要持有 BTree 锁（数据已经快照下来了）
- 即使遍历过程中其他 goroutine 在写，你的迭代器视图不受影响
- 简单，不容易出 bug

**代价**：如果数据量巨大（百万级），一次性收集到切片里内存爆炸。此时应该改用游标式迭代（每次 Next 才读 BTree）。但现阶段数据量不大，这个 trade-off 值。

### IteratorOptions 的设计

两个字段：

- `Prefix`：只遍历前缀匹配的 key。为什么不是 `start/end` range？因为最常见的场景就是「列出某个前缀下的所有 key」，比如 `user:1`、`user:2`。Range 可以后面再加。
- `Reverse`：反向遍历。比如按时间倒序浏览日志。

## 二、ListKeys 和 Fold 的区别

这两个方法看起来很像，实际上侧重点完全不同：

| 方法 | 读什么 | 应用场景 |
|------|--------|----------|
| `ListKeys()` | 只从内存索引拿 key，不碰磁盘 | 管理后台展示 key 列表、统计 key 数量 |
| `Fold(fn)` | 读磁盘拿到完整 value，交给回调处理 | 批量处理全量数据、数据迁移、数据分析 |

一个关键设计决策：`Fold` 的 `fn` 返回 `false` 就提前终止遍历。这让你可以在遍历到某个条件时优雅退出，不用把全量数据都拉出来。

## 三、WriteBatch 的原子性是怎么保证的

### 问题

假设你要执行一组操作：扣 A 的钱，加 B 的钱。如果写到一半宕机了，A 的钱扣了但 B 没加——数据就不一致了。

传统数据库用 WAL（Write-Ahead Log）+ undo log 回滚来处理。我们的思路更简单但同样有效。

### 核心机制：SeqNo + TxnFinished 标记

每个 WriteBatch 分配一个全局递增的 `SeqNo`。Commit 时按这个顺序写：

```
[记录1, SeqNo=N] [记录2, SeqNo=N] ... [TxnFinished, SeqNo=N]
```

关键点：在 TxnFinished 标记写入磁盘之前，内存索引**不动**。

### 崩溃恢复逻辑

启动时遍历所有数据文件，读到有 SeqNo 的记录就按 SeqNo 分组暂存：

```
遇到 SeqNo=N 的记录 → 放到 pendingBatch[N]
遇到 TxnFinished(SeqNo=N) → 把 pendingBatch[N] 全部应用到索引，清空
```

遍历结束后，如果 `pendingBatch` 里还有残留——说明这个 batch 的 TxnFinished 标记没写完，属于**未完成事务**。直接丢弃，后续 merge 的时候磁盘上这些脏数据会被清理掉。

### 为什么用 map 而不是数组存 pendingWrites

`pendingWrites` 用的是 `map[string]*data.LogRecord`，key 是 `string(key)`。

**原因**：同一个 batch 里你可能对同一个 key 写了多次，比如先 Put("a","1") 又 Put("a","2")。用 map 后面覆盖前面，最后 Commit 时只写一次。如果用数组，同一个 key 会写两次，浪费磁盘空间不说，恢复时还要处理覆盖逻辑，麻烦。

## 四、当前的事务隔离级别

### 我们现在用了全局锁

`DB.mu` 是一个 `sync.RWMutex`：
- Put/Get/Delete 用 RLock（Get）或 Lock（写操作）
- WriteBatch.Commit 在写磁盘阶段获取 `db.mu.Lock()`

这意味着：**任何时刻只有一个写操作在执行**。这是「串行化（Serializable）」隔离级别最暴力的实现——全局锁。

好处：实现简单，绝不出现并发异常。
坏处：并发写性能差。两个完全不相关的 key 也得排队。

### 后续可以升级到什么？

规划中的 MVCC（多版本并发控制）可以让读不阻塞写、写不阻塞读。但要搞清楚几个概念的区别：

#### 快照隔离（Snapshot Isolation, SI）

每个事务开始时拿一个快照。读操作都基于这个快照，看不到其他并发事务的修改。事务提交后，修改才对后续事务可见。

SI 解决了：脏读、不可重复读、幻读。

SI 解决不了：**写偏移（Write Skew）**。

#### 什么是写偏移？

经典例子：医生值班系统，要求「至少有一个医生在值班」。

```
Alice 读取：当前值班医生 = {Alice, Bob} → 2人，>= 1 ✓，可以请假
Bob   读取：当前值班医生 = {Alice, Bob} → 2人，>= 1 ✓，可以请假
Alice 更新：把自己标记为休假
Bob   更新：把自己标记为休假
结果：0人值班，约束被破坏！
```

两个事务读的是同一个快照（都有 2 人），各自修改不同行，MVCC 检测不到冲突。这就是写偏移。

只有**串行化（Serializable）**隔离级别才能彻底杜绝写偏移——用 2PL（两阶段锁）或 SSI（Serializable Snapshot Isolation）实现。

#### 为什么不现在实现 MVCC

因为我们索引全在内存。MVCC 意味着同一个 key 在内存里可能存多个版本。简单估算：

如果你有 100 万个 key，开了 10 个并发事务，每个事务都在写不同的数据。极端情况下索引条目膨胀 10 倍——内存直接爆炸。

正确的做法是：把旧版本留在磁盘上，内存索引只指最新版本。读的时候按 SeqNo 去磁盘找对应版本。但这会让读路径变复杂——现在直接 `index.Get(key)` 就能拿到位置，以后还得「在 N 个版本里找可见的那个」。

**结论**：现阶段用全局锁够用，MVCC 等磁盘 merge 和 compaction 做好之后再搞。
