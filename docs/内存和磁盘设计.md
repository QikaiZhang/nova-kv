# 内存与磁盘设计：从第一性原理推导 Bitcask 架构

> 这不是一篇"告诉你 Bitcask 是什么"的文档，而是一篇"带你推导出 Bitcask 为什么是这样"的思考笔记。



> 真正的理解，是能从"我要存数据"这个最朴素的需求出发，一步步推导出每个设计决策的必然性。

---

## 一、从头开始：我要做一个 KV 存储

假设现在什么都没有，我只有一个需求：存一堆 key-value，能存、能取、能删，掉电不能丢。

第一个问题来了：**数据存在哪里？**

### 选项 A：全放内存

- 优点：快，读写都是纳秒级
- 缺点：掉电就没了；内存贵，容量有限

### 选项 B：全放磁盘

- 优点：持久化，容量大，便宜
- 缺点：慢，尤其是随机读写

**结论：必须放磁盘。** 因为"持久化"是存储引擎的基本盘，没有持久化就不叫存储，叫缓存。

好，现在数据确定放磁盘了。

---

## 二、第二个问题：怎么快速找到某个 key？

数据在磁盘上，但磁盘是个大文件，我怎么知道 `user:123` 这个 key 在文件的哪个位置？

### 最笨的办法：从头到尾扫文件

- 优点：简单到不用想
- 缺点：每次查询都扫全文件，数据多了慢到离谱

这显然不行。那怎么办？

→ 你需要一个"目录"，告诉你每个 key 在文件的哪个位置。
→ 这个"目录"，就是**索引**。

> **索引的本质：一个快速查找的目录，key → 数据在磁盘上的位置。**

看到没有？**索引不是凭空出现的，是被"查得慢"这个问题逼出来的。**

很多教程一上来就讲"索引是什么"，但从来不讲"为什么需要索引"。好像索引是天经地义存在的一样。其实不是——索引是为了解决"磁盘上找数据慢"这个问题才诞生的。

---

## 三、第三个问题：索引存在哪里？

现在有了索引这个东西，但它自己存在哪里？这是个关键的设计决策。

### 选项 A：索引也放磁盘

那每次查数据要做两次磁盘 IO：

1. 先查索引（磁盘）→ 找到数据位置
2. 再查数据（磁盘）→ 拿到 value

两次磁盘随机 IO，还是慢。而且索引本身怎么快速查找？索引的索引？索引的索引的索引？→ 无限套娃了。

### 选项 B：索引放内存

- 优点：内存快，查找是内存级别的速度
- 缺点：掉电就没了，重启后索引怎么办？

**但等等——索引丢了可以重建啊！**

因为索引是从数据文件里"提炼"出来的，只要数据文件还在，就能重新把索引建起来。数据是源头，索引是衍生物。目录丢了可以翻书重建，书丢了就真没了。

**这就是 Bitcask 的核心权衡：**

> 用"启动时重建索引"的一次性代价，换取"每次查询都很快"的亿次级收益。

这笔账太划算了。启动慢个几秒算什么？每次查询快几百倍才是真的香。

---

## 四、第四个问题：磁盘上的数据怎么写？

好，现在架构基本定了：

- 数据放磁盘（持久化）
- 索引放内存（快速查找）

那下一个问题来了：磁盘上的数据文件，怎么组织写入？

### 方案一：原地更新（In-Place Update）

找到 key 原来的位置，直接覆盖写。

听起来很直观对吧？新值替换旧值，天经地义。

但现实不会这么友好。我们来一步步拆解问题。

#### 场景 1：新 value 和旧的一样大

```
Offset 100: user2 -> Jack
                ↓
Offset 100: user2 -> Lucy
```

完美，直接覆盖，皆大欢喜。

#### 场景 2：新 value 比旧的大

```
Offset 100: user2 -> Jack (4 字节)
                ↓
Offset 100: user2 -> This is a very very long value... (几十字节)
```

放不下了。怎么办？只有三条路：

**办法一：后面的数据全部往后搬**

把 user3、user4... 全部往后挪，给 user2 腾地方。

这意味着什么？你修改了 50 字节的数据，结果后面 500MB 的数据全部要移动。**这就是写放大（Write Amplification）。**

> **写放大：为了修改很少的数据，却不得不写更多的数据。**

而且更要命的是：所有后面数据的 Offset 全变了，内存里的索引几乎全部失效，要批量更新。这代价大到无法接受。

**办法二：找空洞塞进去**

以前删掉的 key 会留下空洞，新数据找个合适的洞塞进去。

听起来不错，但问题是：空洞是碎片化的。

```
100Byte 空洞 → 300Byte 数据 → 20Byte 空洞 → 500Byte 数据...
```

数据库每天都在干一件事：找哪个洞能放进去？这就是**空闲空间管理（Free Space Management）**。

MySQL 花了几十年都在优化这个问题。复杂度非常高。

**办法三：重新建文件**

把整个文件重新整理一遍，该删的删，该挪的挪。这其实就是 Merge 的雏形，但放在原地更新的语境下，成本太高了，不可能每次写都做。

#### 原地更新的总结

| 问题 | 后果 |
|------|------|
| value 变长 | 写放大，搬数据 |
| 碎片化空洞 | 空闲空间管理复杂 |
| 崩溃恢复 | 写一半断电了怎么办？数据可能损坏 |
| 实现复杂度 | 高，要处理各种边界情况 |

原地更新这条路，越走越重。

---

### 方案二：追加写入（Append Only）

Bitcask 的作者说：我为什么要解决这些麻烦？

```
原来的文件：
  user1 -> Tom
  user2 -> Jack
  user3 -> Alice

现在 Put("user2", "Lucy")，直接追加到末尾：
  user1 -> Tom
  user2 -> Jack
  user3 -> Alice
  user2 -> Lucy  ← 新的写在最后
```

旧数据不管了，就让它躺在那里。内存里的哈希表改一下指针，让 user2 指向最新的位置。完事。

**整个写入路径，永远只有一个动作：Append。**

这一下解决了三个核心问题：

**第一，没有随机写。** 永远写在文件末尾，纯顺序写。SSD 和 HDD 都喜欢顺序写，性能拉满。

**第二，没有搬数据。** 旧数据不要了，以后再说。不会出现"改 50 字节搬 500MB"这种离谱的写放大。

**第三，索引几乎不用改。** 以前原地更新可能导致一堆 Offset 变化，索引要批量更新。现在只改一个 key 的指针，O(1) 搞定。

那代价是什么？

**垃圾越来越多。**

一个 key 被更新了 100 次，磁盘上就有 100 个历史版本，其中 99 个都是垃圾。磁盘空间会越用越大。

这叫**空间放大（Space Amplification）**——实际占用的磁盘空间，比有效数据大很多倍。

> 注意：写放大和空间放大是两回事。
> - 写放大：写一次，实际写入的数据量是多少（关注写入性能）
> - 空间放大：存一份数据，实际占了多少空间（关注存储成本）

Append Only 把写放大降到了极低，但换来了空间放大。又是一个经典的 trade-off。

## 五、Merge：不只是"清理垃圾"

很多人对 Merge 的理解停留在"清理垃圾数据"。这个理解太浅了。

如果面试只说"Merge 就是清理垃圾"，那这个问题你只能拿 30 分。

**Merge 实际上一次解决了三个问题：**

### 1. 空间回收（最直观的作用）

```
Merge 前：
  文件1: [key1=v1][key2=v1][key1=v2][key3=v1]  ← 一堆历史版本
  文件2: [key2=v2][key1=v3]

Merge 过程：
  遍历所有旧文件，只保留每个 key 的最新版本
  写到一个新文件里
  删除旧文件

Merge 后：
  新文件: [key1=v3][key2=v2][key3=v1]  ← 干净、紧凑
```

垃圾没了，空间回收了。这是最容易想到的。

### 2. 提高查询局部性（很多人忽略的点）

Merge 之前，同一个 key 的不同版本散落在各个文件的各个位置。最新版本可能在任何地方。

Merge 之后，所有 key 的最新版本都连续地放在一起。

这意味着什么？**Page Cache 命中率更高。**

操作系统的页缓存是按页加载的。如果热点数据连续分布，一次磁盘 IO 就能加载更多有效数据。如果数据是碎片化的，加载一页可能大部分都是垃圾。

这是 Merge 带来的隐性性能收益。

### 3. 生成 Hint File（Bitcask 的点睛之笔）

Bitcask 在 Merge 的时候，会顺便生成一个 Hint File。里面只存：

```
key | offset | size
key | offset | size
...
```

这东西有什么用？

**启动的时候不用扫描几十 GB 的数据文件了。**

直接读 Hint File，索引秒恢复。因为 Hint File 里只有 key 和位置信息，体积很小，加载飞快。

这是 Merge 顺便完成的工作——反正都要遍历一遍数据，不如顺便把索引信息也整理出来。

### Merge 的本质

所以 Merge 到底是什么？

> **Append Only 将写路径做到极致简单和高效，但代价是产生大量历史版本和墓碑记录，因此后台通过 Merge 完成空间回收、历史版本清理，并重建 Hint File 降低启动恢复成本。这实际上是把复杂度从写路径转移到了后台任务，是典型的"用后台计算换前台写性能"的设计。**

这句话面试说出来，基本就稳了。因为你不是在背"Merge 清理垃圾"，你是在讲设计哲学。

## 六、设计选型的思考框架

学到这里，其实可以提炼出一个通用的思考框架了。

任何存储引擎的设计，本质上都是在这几个维度之间做权衡：

| 维度 | 原地更新 | 追加写入 + Merge |
|------|---------|-----------------|
| 写入性能 | 慢（随机写 + 可能搬数据） | 快（纯顺序写） |
| 空间利用率 | 高（实时回收） | 低（有空间放大） |
| 实现复杂度 | 高（空闲空间管理、崩溃恢复） | 低（写路径极简单） |
| 崩溃恢复 | 复杂（可能写了一半） | 简单（要么写完要么没写） |
| 历史版本 | 没有 | 天然保留（可做快照、时间旅行） |

没有哪个方案是完美的。选哪个，取决于你的场景更在意什么。

- **写多读少，对写入延迟要求极高** → 追加写入（Bitcask、LSM Tree 都是这个思路）
- **读多写少，空间利用率重要** → 原地更新（B+ Tree 系的数据库大多是这个思路）

这就是为什么面试喜欢问"为什么这么设计"——因为它考察的不是你知不知道答案，而是你有没有权衡的思维。

## 七、面试高频问题与回答思路

整理几个这个模块最容易被问到的问题，以及怎么回答才能体现深度。

### Q1：为什么索引要放在内存？不能放磁盘吗？

**回答思路（从三个层次说）：**

1. **性能角度**：磁盘随机 IO 太慢，如果索引也在磁盘，每次查询至少两次随机 IO，性能无法接受。内存查找是纳秒级的，差距几个数量级。

2. **可重建性**：内存索引重启后会丢失，但索引是数据的衍生物，可以从数据文件完整重建。丢了不怕，重建就行。

3. **成本权衡**：重建索引是一次性成本，只在启动时花一次时间。但查询是亿次级成本，每次都要快。用一次启动慢，换每次查询快，这笔账太划算了。

### Q2：为什么索引里存的是"位置"而不是完整的 value？

**回答思路：**

1. **内存容量限制**：value 可能很大（几 KB、几 MB 甚至更大），如果索引存完整 value，内存很快就爆了。索引只存位置信息（十几个字节），内存能装下百万、千万级的 key。

2. **更新成本**：如果索引存 value，每次更新 value 都要改索引。Bitcask 的设计是：写操作只追加到磁盘，然后改一下索引里的指针就行。位置信息很小，改起来极快。

3. **读写分离**：数据文件是只读的（除了追加），不会被修改。只有索引在内存里频繁变化。这样磁盘上的数据文件很"干净"，备份、迁移都方便。

### Q3：为什么用追加写入而不是原地更新？

**回答思路：**

1. **写性能**：追加写入是纯顺序写，磁盘性能拉满。原地更新是随机写，而且可能因为 value 变长导致大规模数据搬移，写放大严重。

2. **实现简单**：追加写入的写路径只有一个动作——往文件末尾写。不用管理空闲空间，不用处理碎片化，崩溃恢复也简单（要么写完要么没写）。

3. **代价是空间放大**：追加写入会产生大量历史版本，空间利用率低。所以需要后台 Merge 来回收空间。这是用空间换时间和简单性的 trade-off。

### Q4：Merge 是做什么的？

**回答思路（千万别说"清理垃圾"就完了）：**

Merge 主要做三件事：

1. **空间回收**：遍历旧文件，只保留每个 key 的最新版本，写到新文件，删除旧文件。回收被历史版本和墓碑占用的空间。

2. **提高局部性**：Merge 后所有最新数据连续存放，Page Cache 命中率更高，读性能更好。

3. **生成 Hint File**：Merge 时顺便生成只包含 key + offset + size 的 Hint File，启动时直接加载，不用扫描全量数据，大幅降低启动恢复成本。

本质上，Merge 是把写路径的复杂度转移到了后台。前台写路径保持极简，后台慢慢收拾。

### Q5：如果内存不够装下所有索引，怎么办？

这是个开放题，没有标准答案，说思路就行。

**可以从这几个方向展开：**

1. **B+ Tree 磁盘索引**：把索引本身也存在磁盘上，用内存缓存热点节点。MySQL InnoDB 就是这个思路。树高低，每个节点对应一个磁盘页，查询时大部分节点在内存缓存里。

2. **LSM Tree**：内存里只放一小部分（MemTable），大部分在磁盘上的 SSTable。写入先写内存，满了刷到磁盘。后台定期合并。牺牲一点读性能，换取极高的写性能。LevelDB、RocksDB 都是这个路线。

3. **分区索引 + LRU 换入换出**：把 key 分成很多区间，每个区间一个索引文件。查询时如果索引不在内存，就从磁盘加载。内存满了就把最久没用的换出去。本质是用户态实现的虚拟内存。

## 八、写在最后

学存储引擎，最忌讳的就是把每个组件当成孤立的知识点来背。

内存索引、追加写入、Merge……这些不是作者拍脑袋想出来的，是被一个个问题一步步逼出来的。

你从"我要存数据"出发，遇到第一个问题"怎么找得快" → 诞生了索引；遇到第二个问题"索引放哪" → 诞生了内存索引；遇到第三个问题"数据怎么写" → 诞生了追加写入；追加写入带来了空间问题 → 诞生了 Merge。

每一步都是问题驱动的，每一个设计决策背后都有 trade-off。

能看到这一层，你就不是在"学知识"，而是在"练思维"。面试的时候，面试官问的从来不是"你知道多少"，而是"你怎么想"。
