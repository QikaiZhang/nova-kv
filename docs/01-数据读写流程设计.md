# 数据读写流程设计与实现

## 一、核心设计思想

### Bitcask 模型本质
基于 **Append-only Log + 内存索引** 的存储模型：
- **磁盘**：真相来源（Source of Truth），所有写入以追加日志形式持久化
- **内存**：只存索引（Key → FileID + Offset），不存实际数据
- **崩溃恢复**：扫描磁盘文件即可重建内存索引，数据不丢

### 写入顺序：先磁盘，后索引
这是 Bitcask 最关键的设计决策：
1. 写磁盘成功 → 数据已持久化
2. 再更新内存索引

**为什么不能反过来？**
- 如果先写索引再写磁盘，写磁盘失败时索引指向了不存在的数据 → 数据不一致
- 先写磁盘后写索引，即使索引更新失败，重启扫描磁盘仍能恢复 → 最终一致

## 二、初始理解的偏差与纠正

### ❌ 偏差 1：LogRecord 的生命周期
> 原理解："用户发请求到内存开始有，进入磁盘之前消失"

**纠正**：
- LogRecord 有两种形态：
  - **内存形态**：结构体，用于构造和编码，写入后可释放
  - **磁盘形态**：编码后的二进制字节，**永久存在**于数据文件中
- 即使 key 被覆盖/删除，旧记录仍然躺在磁盘上，只是索引不再指向它
- 这也是为什么后续需要 Merge 压缩来回收空间

### ❌ 偏差 2：切换文件时"对旧文件做持久化"
> 原理解："超过了对旧的日志文件做持久化"

**纠正**：
- 旧文件本来就已经在磁盘上了，不是"做持久化"
- 切换活跃文件时实际做的三件事：
  1. **Sync 刷盘**：确保内核缓冲区数据真正落到磁盘
  2. **Close**：关闭文件描述符
  3. **标记 Immutable**：从 activeFile 移入 olderFiles 集合

### ❌ 偏差 3：读的错误分类
> 原理解："读的错误有 key 不存在和 key 对应的文件不存在以及写入失败三种"

**纠正**：写入失败不是读的错误。读路径完整错误分类：
1. Key 不存在（索引查不到）
2. 文件打开失败（异常情况）
3. 读取失败（Seek/Read 出错）
4. CRC 校验失败（数据损坏，工程必加）
5. 记录已被删除（读到墓碑记录）

## 三、LogRecord 结构设计

### 磁盘编码格式

```
+-------+--------+-----------+-------------+-----+-------+
| CRC   |  Type  |  KeySize  |  ValueSize  | Key | Value |
+-------+--------+-----------+-------------+-----+-------+
  4B      1B       变长(4B)    变长(4B)       变长   变长
```

### 字段说明

| 字段 | 大小 | 作用 |
|------|------|------|
| CRC | 4字节 | 数据完整性校验，防止磁盘静默损坏 |
| Type | 1字节 | 区分 Normal（正常写入）/ Deleted（墓碑删除） |
| KeySize | 变长 | Key 的字节长度，变长编码节省空间 |
| ValueSize | 变长 | Value 的字节长度 |
| Key | 变长 | 实际的 key 数据 |
| Value | 变长 | 实际的 value 数据 |

### 课程方案 vs 线上方案

| 维度 | 课程方案 | 线上方案 |
|------|----------|----------|
| CRC 校验 | 可省略，降低复杂度 | **必须有**，磁盘静默损坏是真实风险 |
| 长度编码 | 固定 4 字节 uint32 | 变长 Varint，小 key/value 节省空间 |
| 墓碑标记 | 单独的删除记录类型 | 同样的设计，Type 字段区分 |

## 四、DataFile 设计

### Active File vs Immutable File

| 特性 | Active File | Immutable File |
|------|-------------|----------------|
| 可写 | ✅ 是 | ❌ 否 |
| 可读 | ✅ 是 | ✅ 是 |
| 数量 | 只有 1 个 | 可以有 N 个 |
| 读路径感知 | ❌ 不感知 | ❌ 不感知 |

**设计亮点：读路径完全不关心文件状态**
- 读的时候只认索引里的 (FileID, Offset)
- 不管是 active 还是 older 文件，读取方式完全一样
- 读写职责分离，实现简单，正确性容易保证

### 文件切换逻辑

```
写入新记录前检查：
  当前 WriteOff + 本次记录大小 > 阈值 ?
    ├─ 否 → 直接追加
    └─ 是 → Sync 旧文件 → 移入 olderFiles → 新建 active file
```

**关键点**：预判写入后的大小，而不是写完再判断。保证单条记录不会跨文件。

## 五、写入主流程

### Put 完整流程

```
Put(key, value):
  1. 参数校验（key 非空）
  2. 构造 LogRecord{Key, Value, Normal}
  3. 检查 active file 是否能容纳
     - 不能容纳：Sync 旧文件 → 新建 active → 旧文件归档
  4. 编码 LogRecord → 二进制 buf
  5. 记录当前 WriteOff 作为本次 offset
  6. 写入 active file
  7. SyncWrites 配置决定是否立即刷盘
  8. 更新内存索引：index.Put(key, Pos{Fid, Offset, Size})
  9. 返回
```

### 关键问题：写磁盘成功，索引更新失败怎么办？

**答案：可以接受，最终一致。**
- 后果：暂时读不到这条数据（索引里没有）
- 恢复：重启后扫描磁盘重建索引，数据会找回来
- 磁盘是真相来源，索引只是缓存

## 六、读取主流程

### Get 完整流程

```
Get(key):
  1. 参数校验
  2. 查内存索引 → (FileID, Offset)
     - 查不到 → ErrKeyNotFound
  3. 根据 FileID 找到 DataFile（active 或 older）
  4. ReadLogRecord(Offset) → 解码出 LogRecord
  5. 判断 Type：
     - Normal → 返回 Value
     - Deleted → 返回 ErrKeyNotFound（墓碑）
```

## 七、工程实现要点

### 代码结构

```
nova-kv/
├── data/
│   ├── log_records.go   # LogRecord 结构 + 编解码
│   └── data_file.go     # DataFile 读写封装
├── index/
│   ├── index.go         # Indexer 接口定义
│   └── btree.go         # BTree 索引实现
├── fio/
│   ├── io_manager.go    # IOManager 接口
│   └── file_io.go       # 标准文件 IO 实现
├── db.go                # DB 主结构体 + Put/Get/Delete
├── options.go           # 配置项
└── errors.go            # 错误定义
```

### 依赖注入与可扩展性

1. **IOManager 接口**：底层 IO 可替换（文件 IO / MMap / 等）
2. **Indexer 接口**：索引实现可替换（BTree / ART / 跳表 / 等）
3. **Options 配置化**：数据目录、文件大小、刷盘策略均可配

### 并发安全
- DB 级别一把 `sync.RWMutex`
- 写操作（Put/Delete）：写锁
- 读操作（Get）：读锁
- 索引内部自己也有锁，双层保护

## 八、面试价值 ⭐⭐⭐⭐⭐ 高频

### 为什么面试官喜欢问 Bitcask？

1. **存储模型多样性**：除了 LSM，是否了解其他存储引擎设计
2. **磁盘 IO 理解**：顺序写 vs 随机写的性能差异
3. **崩溃恢复**：分布式存储必考点，如何保证数据不丢
4. **设计权衡**：写入性能 vs 空间放大（Merge 回收）
5. **读写分离思想**：职责分离如何简化实现、保证正确性

### 项目里怎么介绍

> "采用 Bitcask 存储模型，基于 Append-only 顺序写保证高写入吞吐，内存索引（BTree）保证 O(logN) 读性能。写入路径遵循先持久化磁盘再更新内存索引的原则，崩溃后通过扫描数据文件重建索引，保证数据不丢失。活跃文件与不可变文件分离设计，读路径无需关心文件状态，简化了实现复杂度。"

### 值得写博客的点

1. **Bitcask 读写流程图**：写路径（追加日志 + 更新索引）vs 读路径（查索引 + 随机读）
2. **LogRecord 二进制编码格式详解**：为什么用变长编码、CRC 的作用
3. **崩溃恢复机制**：从磁盘文件重建索引的完整过程
4. **空间放大与 Merge 压缩**：Append-only 的代价与解决方案

### 值得画架构图的点

1. 整体架构图：内存索引 + 磁盘数据文件的分层
2. 写入时序图：Put 操作的完整调用链
3. 文件状态机：Active → Immutable 的转换
4. 启动恢复流程图：加载文件 → 构建索引
